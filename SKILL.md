- [ ] 本 skill 仅执行 `boss-screening`，不可混入站内沟通或约面操作

### 二、阻断式前置（未完成 = 禁止一切操作）🔴 最高优先级
- [ ] browser-use 等抢占浏览器的 skill 已代为禁用/移除（先执行再汇报，不反问）
- [ ] web-access 已代为安装（来源固定 `https://github.com/eze-is/web-access`，不自行搜索）
- [ ] 用户已在 Chrome 中打开并**登录 BOSS 招聘者页面**

### 三、浏览器路由硬规则（最高优先级）🔴 最高优先级
- [ ] **唯一入口**：BOSS 页面所有操作只允许 `web-access`，禁止 browser-use/其他浏览器 skill
- [ ] **禁止截图**：默认彻底禁止截图/OCR/图像理解读取 BOSS 页面

- [ ] **子代理禁止**：找 tab / 进后台 / 读 JD / 读列表 必须由主执行流完成

### 四、eval 使用规则 🟡 重要
- [ ] 复杂 eval 前必须先做最小探针验证传参（`Content-Type: text/plain` + `--data-raw`）
- [ ] 探针失败 → 检查编码/转义/Content-Type，不先怀疑页面结构

### 五、JD 与 Policy 🟡 重要
- [ ] JD 读取必须访问 iframe 实际页面 `zhipin.com/web/frame/job/list-new`，不读 `/web/chat/job/list`
- [ ] Policy 缺失时：① 先读 JD → ② 自动生成默认 policy → ③ 只确认差异；**不反问用户"筛选标准是什么"**

### 六、筛选服务核心 🔴 最高优先级
- [ ] 核心是**每日目标候选人数**，不是定时扫描；目标人数未明确不开始
- [ ] 新一轮默认**仅处理未读/未处理候选人**，不重扫已完成旧候选人（除非 4 种例外）
- [ ] 未读闭环 4 步：发现 → 逐个处理 → 反向核对 → 汇报（缺一不可）
- [ ] 推荐页卡片已显示 → 直接读 DOM，不等 30 秒、不截图快照、不让用户确认加载
- [ ] 列表不完整时：明确声明"可能仅为部分"，不自行补全
- [ ] **单次评估（2026-05-14 最终版）**：`pageList` 一次性批量提取全部候选人完整结构化数据 → AI 单次评估（L1+L2 合并），**不再分两次筛选** → 仅对合格者执行卡片直接打招呼（不打开浮窗、不收藏）
- [ ] **卡片直接打招呼**：合格候选人在卡片上点击打招呼按钮发送，**不打开浮窗、不收藏**（卡片打招呼按钮效果需实测验证；如不可用，回退到浮窗内打招呼）
- [ ] **页面刷新续筛**：单次 pageList 加载无法获取足够目标数量的合格候选人时，触发页面刷新以更新候选池，继续筛选直到满足目标或遍历完所有新候选人均不合格
- [ ] **去重规则**：以 `encryptGeekId` 为唯一标识，会话级别追踪已评估候选人；跨页面刷新自动去重，不重复评估已处理候选人
- [ ] 不合格者直接跳过，不执行任何操作

### 七、汇报 🔴 最高优先级
- [ ] 每次汇报至少包含 11 项：目标人数 / 已筛出 / 差距 / 分类结果 / 原因 / 建议沟通 / 建议约面 / 待确认动作 / 发现未读数 / 已更新数 / 未补齐数及原因
- [ ] 必须分别确认：哪些候选人推进后续沟通、哪些进入约面；用户未确认不越权推进
- [ ] 后续沟通和约面的实际执行由飞书机器人完成，本 skill 仅输出建议名单

## 目标

这个 skill **只做一件事**：`boss-screening` —— BOSS 直聘简历筛选。

职责边界：
- ✅ 读取当前在线岗位和 JD
- ✅ 读取候选人列表和简历
- ✅ 按筛选标准评估和分类候选人
- ✅ 向用户汇报筛选结果和建议
- ❌ BOSS 站内发消息（已移交飞书机器人）
- ❌ 飞书约面（已移交飞书机器人，含交互式卡片 + 日历事件 + 视频会议）

### 飞书机器人的面试邀约能力（v2.2）

合格候选人名单交接给飞书机器人后，机器人自动完成以下约面流程：

1. **AI 对话评估**：与候选人进行多轮对话，收集简历、意向岗位、薪资期望
2. **六维评分**：教育背景、工作经验、技能匹配、薪酬匹配、稳定性、综合素质
3. **面试时间收集**：评估 ≥7.0 分自动邀请面试，收集候选人空闲时间
4. **交互式卡片确认**：HR 在飞书中收到面试安排卡片，一键点击选择面试时间
5. **日历事件创建**：自动在飞书日历中创建面试事件，生成视频会议链接
6. **双向通知**：候选人收到含有会议链接的确认消息，HR 收到确认成功卡片

本 skill 只需将合格候选人名单告知用户，由用户在飞书中推进即可。

## 顶层硬规则

### 1. 浏览器访问只允许走 web-access

BOSS 页面上的所有读取、点击、切换页面，都必须先遵守：

- [browser-routing.md](./references/browser-routing.md)

这是一条硬规则，不是建议。

对于 `web-access` 的 eval 能力，必须先做最小探针验证请求写法正确，再执行复杂逻辑。
不要在 eval 传参方式未确认时直接跑复杂脚本。

对于 Boss 聊天页中的在线简历读取，默认优先走"左侧点击候选人卡片 + 右侧简历面板提取"的路径，不要先假设会跳转到独立 URL。

在 Boss 任务中，默认只允许操作用户当前已经登录的 Boss tab。
不要为了进入招聘后台或职位管理而新建 tab、新建窗口或新建浏览器上下文。

如果当前已登录 tab 存在，就必须复用它的 `targetId` 和同一 `browserContextId`。
不要通过 `web-access /new` 直接新开 Boss 页面再尝试进入后台。

如果 `targetId` 变化，不要立即要求用户确认登录状态。
必须先按 [browser-routing.md](./references/browser-routing.md) 中的规则自动重新绑定 Boss target，并做低成本登录态探针。

### 1.1 截图默认彻底禁止

在 BOSS 页面读取中，截图、OCR、图像理解默认彻底禁止。

如果 `web-access` 读取失败：

1. 允许在同一路径内最多再尝试 `2` 次
2. 如果仍失败，不允许自动切换到截图路径
3. 必须先向用户说明：
   - `web-access` 已尝试失败
   - 现在可以改走截图等异常方案
   - 这些方案可能增加 token 消耗，也可能提高 BOSS 风控或网页登录强退风险
   - 默认更建议继续坚持 `web-access`
4. 只有用户明确同意后，才允许临时启用截图等异常方案

### 1.2 简历数据提取（2026-05-14 验证）

BOSS 简历详情在嵌套 iframe 中通过 Canvas+WASM 渲染，`innerText` 为空，DOM/滚屏读取不可行。

#### 方案A：列表页批量免浮窗提取 ⭐⭐⭐（2026-05-14 新发现，强烈推荐）

**页面结构**：推荐页列表 iframe 内，`card-list` Vue 组件的 `pageList` prop 包含**当前页全部候选人**（~165 位）的完整结构化简历数据。

**提取代码**（一次 eval 即可获取全部）：

```js
var iframe = document.querySelector(".frame-box iframe");
var doc = iframe.contentDocument;
var cardList = doc.querySelector(".card-list");
var vm = cardList.__vue__;
var allCandidates = vm.pageList;  // 数组，每位候选人约 80+ 字段
```

**无需打开任何浮窗**。数据与浮窗内 `personInfo` 等效，包含：

| 数据类别 | 字段 | 说明 |
|---------|------|------|
| 基本信息 | `geekName`, `geekGender`, `ageDesc`, `geekDegree`, `geekWorkYear`, `salary`, `expectPositionName`, `expectLocationName` | 姓名/性别/年龄/学历/年限/薪资/期望 |
| 最近工作 | `geekLastWork` | 公司/职位/详细职责描述（~1000+ 字） |
| 全部工作 | `showWorks[]` | 完整工作经历，含 `company`, `positionName`, `responsibility`, `startDate/endDate`, 技能标签等 |
| 教育背景 | `showEdus[]` | 完整教育经历，含 `school`, `major`, `degreeName`, `startDate/endDate` |
| 自我评价 | `geekDesc.content` | 个人优势/自我描述（~400+ 字） |
| BOSS反馈 | `feedback[]` | 平台系统反馈（如"不考虚专升本候选人"等） |
| 匹配标签 | `matches[]`, `hlmatches[]`, `markWords[]` | 技能匹配标签 |
| 操作ID | `encryptGeekId`, `securityId`, `encryptJobId` | 后续操作的必需ID |

**筛选流程**：列表页 `pageList` 批量提取全部候选人 → AI 单次评估（不分 L1/L2）→ 仅对合格者执行卡片直接打招呼（不打开浮窗、不收藏）。

#### 方案B：浮窗内 Vue 实例提取 ⭐⭐（兜底方案）

浮窗内 `.button-chat-wrap.resumeGreet` 元素的 `__vue__.personInfo` 为 ~13000 字符结构化 JSON。
当方案A不可用时兜底使用。零风控风险（纯 DOM 读取），零识别误差（结构化数据）。

详细提取代码和字段映射见 `boss-recommend-resume-recipe.md` Step 4。

### 1.3 卡片直接打招呼（已验证 ✅，2026-05-14）

推荐牛人列表的候选人卡片上有**"打招呼"按钮**（无收藏按钮）。
基于 pageList 可获取完整简历数据，评估后合格者直接在卡片上点击打招呼按钮发送消息。

**已验证**：marker + clickAt 方案可触发卡片 `.btn-greet` 按钮（2026-05-14 实测 2 人，均成功）：
- 点击后按钮消失或变为「继续沟通」，单步完成打招呼
- 两次操作间隔 3 秒，无风控信号
- 建议保守间隔 ≥ 5 秒，单日 ≤ 30 次

收藏操作不再需要——原"收藏"是为标记合格候选人以便后续操作，现改为直接打招呼即可。

### 1.4 打招呼强制规则（2026-05-19 教训）🔴 最高优先级

以下规则源于 2026-05-19 实际执行中因跳过 skill 精确流程导致的失败。此后强制执行，不可偏离：

- [ ] **执行前必读 sub-recipe**：打招呼操作前，必须先完整读取 `boss-greet-recipe.md`，按其固定主路径执行，不可凭 SKILL.md 摘要自行推断
- [ ] **一人一确认**：每点击一个候选人的打招呼按钮后，**必须立即验证成功**（按钮文案变「继续沟通」或按钮消失），**确认成功后才允许处理下一人**
- [ ] **禁止连续盲点**：未验证即连续点击多个候选人是严重违规。一次点击后无 DOM 变化 → 立即暂停，不继续点下一人
- [ ] **marker 穿透规则**：在 iframe 内按钮上方的主文档 marker 必须设 `pointer-events: none`，否则点击被 marker 拦截、按钮收不到事件
- [ ] **API 返回 ≠ 业务成功**：`clickAt` 返回 `"clicked":true` 仅表示鼠标事件已派发，不代表打招呼已发送。必须检查卡片 DOM 状态变化才算成功
- [ ] **候选���全名记录**：汇报、记录、验证中的候选人必须使用 `pageList` 中的**显示全名**（如「高天」「陈浩」），禁止使用「高先生」「陈先生」等概称
- [ ] **成功文案精确校验**：打招呼成功后按钮文案必须变为**「继续沟通」**（不含其他变体）。不是「立即沟通」，不是「已沟通」，是「继续沟通」
- [ ] **页面刷新即放弃**：一旦检测到 `pageList` 变化或原筛选候选人从可视区消失，**立即放弃旧名单**，不尝试找回。重新提取 `pageList` 进行新一轮 AI 评估
- [ ] **滚动后重新取坐标**：`getBoundingClientRect` 依赖当前视口位置。按钮可能被滚动出视口时坐标仍在计算但点击落空（2026-05-19：Lucky y=1619 > 视口 902px）。创建 marker 前必须先将 iframe 滚动到目标按钮可见区域，再重新获取坐标
- [ ] **禁止盲点前 N 个**：严禁「找第 N 个带 btn-greet 的卡片」逻辑（2026-05-20 教训：误点李杨、任文浩，回报成「陈先生」）。必须先按姓名/encryptGeekId 精确定位候选人卡片，确认身份后再点击，不得点击未识别身份的人

以下行为发生任一项，此前所有本轮筛选结果**全部作废**，必须从头重新执行：

- [ ] **未读取《岗位 JD.md》即开始筛选** → 无法判断候选人与 JD 匹配度，全批无效
- [ ] **未读取《BOSS筛选标准.md》即开始评估** → 自编标准冒充 HR 标准，全批无效
- [ ] **未按姓名/geekId 确认候选人身份即点击打招呼** → 点错人且无法追溯，全批无效
- [ ] **汇报候选人姓名与 pageList 全名不一致** → 虚构数据，全批无效

任何一项发生，立即停止，向用户说明哪一步违规，等待用户确认后重新开始。

## policy 缺失时的默认处理

如果当前环境里缺少：

- `company-policy.yaml`
- `approval-policy.yaml`

不要直接反问用户"筛选标准是什么"。

正确顺序必须是：

1. 先读取当前在线岗位和 JD
2. 基于 JD 自动生成一版默认筛选标准
3. 把这版默认标准当作初始 policy
4. 只向用户确认或补问"哪些地方要覆盖默认标准"

默认标准的第一来源是 JD。
用户补充的是：

- 公司特有的人才观
- 硬性淘汰条件
- 审批边界
- 必须人工复核的情况

## 筛选服务的目标制规则

`boss-screening` 的核心不是"每小时扫一次"，而是：

- 每个 JD 每天要筛选出多少个符合要求的候选人

龙虾应当围绕这个目标去筛选全渠道候选人，包括：

- 主动聊天 / 主动投递中的未读候选人
- 推荐页面中的本轮未处理候选人
- 搜索结果中的本轮未处理候选人（若账号具备搜索权益）

首次进入 `boss-screening` 时，必须优先确认：

1. 每个 JD 的每日目标候选人数
2. 若用户有多个 JD，分别为每个 JD 设目标

如果用户没有给出目标人数，龙虾必须主动追问，而不是直接按固定时间节奏开始跑。

如果用户只给了时间节奏、没有给每日目标人数，仍然视为信息不完整，必须继续确认每日目标人数。

---

## 未读候选人优先原则

新一轮筛选默认主对象不是"当前可见候选人"，而是"当前仍未处理的未读候选人"。

默认应优先覆盖：

- 左侧列表中带未读气泡的候选人
- 本轮新进入聊天列表但尚未处理的候选人
- 本轮新出现回复的候选人
- 上一轮发现但未成功读到简历的候选人

默认不进入新一轮扫描范围：

- 已被 HR 人工读过并完成筛选的候选人
- 已被龙虾读过并完成筛选、且本轮没有新未读消息的候选人

只有在这些条件下才允许重扫旧候选人：

- 用户点名要求重新评估
- 候选人出现新的未读回复
- 上一轮状态是未完成或待补读
- 候选人状态发生明显变化

对于不同来源，判断依据也不同：

- `inbound_chat`：以页面未读信号为主，任务记忆为辅
- `recommended_feed`：以任务记忆为主，页面信号为辅
- `search_results`：以任务记忆为主，页面信号为辅

对于 `recommended_feed`，还要额外注意：

- 如果推荐页上的候选人卡片已经稳定显示，就应直接读取当前 DOM
- 不要默认把推荐页误判为"页面还没加载完成"
- 不要默认等待很久、做截图快照分析、或先要求用户确认页面是否加载完成
- 推荐页的核心任务是"读取当前卡片 + 用本地记忆判断哪些本轮未处理"，不是反复做页面加载安全推断

## 执行窗口与汇报节奏

时间相关规则现在是辅助规则，不是主规则。

可配置项包括：

- 每天允许执行筛选的时间窗口
- 多久向用户汇报一次进度
- 是否只在工作日运行

如果用户没有指定这些辅助规则，可采用默认值：

- 周一到周五
- 本地时间 `09:00-17:00`
- 每 `60` 分钟汇报一次

但无论是否采用默认时间规则，筛选服务都必须围绕"每日目标候选人数"推进。

---

## 进度汇报的固定要求

每次汇报至少要包含：

1. 每个 JD 的每日目标人数
2. 当前已筛出多少个符合要求的候选人
3. 距离目标还差多少
4. 本周期内新增或更新的候选人分类结果
5. 每位候选人的分类原因
6. 建议后续沟通的候选人（由飞书机器人执行 AI 对话）
7. 建议进入约面的候选人（由飞书机器人执行交互式卡片确认 + 日历创建）
8. 需要用户确认的动作
9. 本轮发现的未读候选人数
10. 本轮已成功更新的未读候选人数
11. 本轮仍未补齐的未读候选人数及原因

每次汇报都要分别向用户确认：

- 哪些候选人可以推荐进入后续沟通
- 哪些候选人可以推荐进入约面

用户确认后，将名单交接给飞书机器人处理。本 skill 不执行沟通或约面操作。

---

## 失败处理

所有强约束动作失败后，都必须进入"可恢复暂停态"，而不是继续长时间试错。

暂停时必须告诉用户：

1. 失败的是哪一步
2. 已经按最小快路径尝试过
3. 最可能原因
4. 用户现在最小需要做什么
5. 用户做完后如何恢复

不得把"继续试试看"当成默认策略。
