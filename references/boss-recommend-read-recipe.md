# BOSS 推荐页读取固定方案

## 定位

这个文档用于 `recommended_feed`。
目标不是“判断页面是不是神秘地还没加载完”，而是以最低 token 成本确认：

1. 当前是否已经进入推荐页
2. 当前页是否已经存在可读取的候选人卡片
3. 当前卡片里哪些候选人属于本轮未处理对象

## 读取原则

只要推荐页上已经稳定显示候选人卡片，就视为页面可读取。

此时必须：

1. 直接读取 DOM
2. 直接提取候选人卡片文本和稳定标识
3. 结合本地任务记忆判断哪些候选人属于本轮未处理

不要默认：

- 等待 30 秒
- 让用户确认“页面是否加载完成”
- 让用户描述页面内容
- 默认退回截图快照分析
- 用“可能需要手动触发加载”作为第一判断

## 固定主路径

1. 确认当前 target 已在 Boss 推荐页
2. 用短 selector 检查候选人卡片容器是否存在
3. 用短 eval 统计：
   - 卡片数量
   - 每张卡片的姓名 / 职位 / 期望 / 标签等可见文本
   - 是否存在“加载更多”或继续滚动信号
4. 若卡片数量大于 0：
   - 直接进入候选人卡片提取
   - 基于本地记忆过滤本轮未处理候选人
5. 若卡片数量等于 0：
   - 再做一次低成本 DOM 分层探针
   - 只检查容器、iframe、懒加载占位、空列表文案、登录失效信号
6. 只有两轮 DOM 探针都无法确认页面状态时，才允许暂停并告诉用户当前卡在推荐页读取

## 低成本 DOM 探针

探针顺序固定为：

1. 页面标题和 URL 是否符合推荐页
2. 主内容容器是否存在
3. 候选人卡片节点数量是否大于 0
4. 是否存在“暂无结果 / 没有更多 / 加载中 / 请登录 / 风控验证”之类文案
5. 是否存在 iframe，且候选人内容是否实际在 iframe 内

探针输出必须优先结构化，例如：

- `is_recommend_page=true`
- `card_count=12`
- `has_load_more=false`
- `empty_state=false`
- `risk_prompt=false`

不要输出一大段泛化推测。

## 增量读取规则

只有在页面明确存在下面信号之一时，才继续增量读取：

- “加载更多”按钮
- 继续滚动后出现新增卡片
- 容器滚动高度继续增长

如果没有这些明确信号，就把当前已显示卡片视为本轮可直接处理集合。

## 暂停条件

只有在下面任一情况出现时，才允许暂停并请求用户帮助：

1. 推荐页 DOM 两轮探针后仍无法判断页面状态
2. 明确出现登录失效、验证码、风控验证
3. 明确出现空列表，但与用户实际页面明显矛盾

暂停时要告诉用户的是：

- 当前停在“推荐页 DOM 无法确认”
- 已完成的探针结果是什么
- 下一步最小恢复动作是什么

不要要求用户重新描述完整页面内容。
