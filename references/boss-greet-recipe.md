# BOSS 推荐页打招呼固定方案

## 定位

这个文档只处理“在推荐牛人页对已批准候选人发送首条打招呼”。

目标是保持路径短、定位准、确认快。
不要把它扩展成通用浏览器调试、事件系统研究或前台鼠标控制方案。

## 重要事实：权益优先

推荐牛人页打招呼可能受账号当日沟通权益限制。

真实测试发现：当账号当天只剩 3 次推荐牛人沟通权益时，前 3 位候选人可以成功，后续候选人无论自动点击还是用户手动点击都没有反应。

因此，出现“前几个成功、后续连续失败”时，第一优先级不是怀疑 selector、Vue 事件或 click 方式，而是先判断：

- 当日推荐牛人沟通权益是否已耗尽
- 页面是否存在剩余次数、权益、沟通额度、开通权益、升级套餐等提示
- 手动点击是否同样无反应

如果用户手动点击也无反应，或页面有明确权益/额度限制信号，应进入：

- `paused_for_boss_contact_quota_exhausted`

并记录：

- 已成功打招呼人数
- 本轮剩余未打招呼候选人
- 预计可在明日权益恢复或购买/开通权益后继续

不要继续测试更多 click 方式。

## 前置条件

开始打招呼前，必须同时满足：

1. 候选人已经通过筛选，或已获用户批准进入沟通
2. 当前 Boss 登录态稳定
3. 当前任务正在使用 `web-access`
4. 当前推荐页候选人卡片已稳定显示
5. 当前话术已经准备好，不在页面里边想边写

## 固定主路径

1. 在当前推荐页定位目标候选人卡片
2. 用稳定标识绑定“候选人卡片”和“该卡片内的打招呼按钮”
3. 做候选人身份确认
4. 点击该候选人卡片内已绑定的打招呼按钮
5. 在当前页面完成成功确认
6. 成功后记录发送结果和发送时间
7. 再进入下一位候选人

这是一条“一人一事务”路径。
不要先识别一批候选人，再回头按按钮序号批量点击。

## 稳定定位规则

推荐页打招呼时，默认禁止使用纯位置索引定位，例如：

- XPath 第 `N` 个按钮
- 第 `N` 张卡片里的按钮
- `button[4]`
- “当前列表第 2 个/第 4 个按钮”

必须优先使用：

1. 候选人唯一 ID
2. 卡片节点上的稳定属性，例如 `data-id`、`href`、业务主键
3. 候选人姓名 + 职位 + 当前卡片文本上下文的组合绑定
4. 如果页面没有唯一 ID，则用 eval 给目标卡片或目标按钮临时写入唯一 `data-*` 标记

也就是说，先锁定“这是哪位候选人”，再锁定“这个按钮属于这位候选人”。
不要倒过来先点按钮，再猜它属于谁。

## 无唯一 ID 时的临时标记方案

推荐页卡片可能没有稳定唯一 ID，但卡片内部通常包含候选人姓名和打招呼按钮。

这种情况下，不要退回 XPath、nth-child 或“第 N 个按钮”。
必须使用“先按文本找到卡片，再给卡片内按钮临时打标”的方式。

固定步骤：

1. 用 eval 在当前 DOM 中查找包含目标候选人姓名和必要上下文的卡片
2. 在该卡片内部查找打招呼按钮
3. 给该按钮写入临时属性，例如：
   - `data-lobster-greet-target="candidate_name_or_hash"`
4. 再用 CSS selector 调用 `web-access`：
   - `[data-lobster-greet-target="candidate_name_or_hash"]`
5. 点击完成后清理或覆盖该临时标记

这样做的目的：

- 仍然让 `/clickAt` 或 `/click` 使用 CSS selector
- 避免 `querySelector` 不支持 `:contains`
- 避免 XPath / nth-child 在虚拟滚动后失真
- 避免误点到其他候选人的打招呼按钮

如果候选人同名，必须加入职位、年龄、学校、期望岗位等卡片上下文共同匹配。
如果仍无法唯一匹配，则进入 `paused_for_candidate_binding`，不要继续点按钮。

## 点击路径

推荐页打招呼按钮默认只允许按下面顺序尝试：

1. `web-access /clickAt`，使用目标按钮 CSS selector
2. `web-access /click`，使用目标按钮 CSS selector
3. 单次 DOM `element.click()` 兜底

任何一步成功后都要立即停止继续试其他点击变体。

不要引入：

- 截图坐标点击
- browser-use
- 额外 RPA 工具
- 长链 Vue 事件研究

如果这三类点击都没有让当前卡片状态变化，先进入“权益耗尽优先诊断”，不要继续扩展点击方式。

## 成功确认

成功确认优先级固定为：

1. 当前候选人卡片内按钮文案是否变化
   - 例如从“立即沟通 / 打招呼”变成“继续沟通 / 已沟通”
2. 当前候选人卡片操作区是否变化
   - 例如按钮消失、按钮禁用、按钮样式变化、操作区被替换
3. 当前候选人卡片主状态文案是否变化
   - 例如新增“已联系”“已沟通”“继续沟通”
4. 页面是否出现短时成功提示
   - 例如 toast、轻提示、顶部提示条
5. 当页或全局打招呼计数是否增加

不要只因为“点击 API 返回成功”就当作发送成功。
也不要只因为“全局计数增加”就确认目标候选人成功，必须优先看目标卡片自身状态。

## 权益耗尽优先诊断

如果出现下面模式，应优先判断为权益耗尽或账号权益限制：

1. 前 1 到 3 个候选人打招呼成功
2. 后续多个候选人连续点击无效
3. 目标候选人绑定准确
4. 页面没有明显技术错误
5. 用户手动点击也无反应，或页面存在权益/额度提示

此时进入：

- `paused_for_boss_contact_quota_exhausted`

并停止本轮推荐页打招呼。

暂停汇报应包含：

- 今日已成功沟通人数
- 当前疑似权益上限
- 尚未触达但已筛选通过的候选人名单
- 建议明日权益恢复后继续，或让用户开通/购买更多沟通权益

不要把这类情况误判为：

- selector 失效
- Vue 事件失败
- web-access 不可用
- 需要前台鼠标点击

## 外部确认的边界

推荐页打招呼默认不跳聊天页确认。

只有当前页没有足够成功信号，且没有权益耗尽迹象时，才允许做一次外部确认。

如果必须离开推荐页做外部确认：

1. 只允许做一次
2. 确认完成后，如果还要继续处理下一位候选人，必须重新回到推荐页重新定位
3. 回到推荐页后，禁止复用上一轮的 XPath 索引、按钮序号、卡片序号

## 单轮发送上限

打招呼默认必须分轮执行。

单轮内必须遵守：

1. 小批量发送
2. 相邻发送之间加入随机短暂停顿
3. 每发送 2 到 3 人，做一次轻量状态校验
4. 单轮达到权益上限、用户设定上限或异常上限后立即暂停并汇报

不要因为同一套路径刚刚成功，就无限连续发送。

## token 经济性要求

推荐页打招呼默认要优先节省 token：

1. 优先在单个 eval 中提取“候选人标识 + 卡片文本 + 按钮状态”
2. 优先在当前页完成成功确认，减少 `navigate`
3. 不要为每位候选人重复跑整页深度分析
4. 每位候选人只允许一次快速补试
5. 失败后输出结构化结果，不输出长篇泛化原因分析
6. 连续失败时优先进入权益耗尽诊断，不继续发散 click 方案

## 禁止项

以下行为不允许作为默认打招呼路径：

- 一边看页面一边临时生成长话术
- 起子代理研究 Vue 或前端事件
- 反复切换 DOM click / 坐标点击 / 截图方案
- 发送失败后长时间停在同一候选人页面试错
- 要求用户手动帮忙点击候选人或手动发送
- 用 XPath / DOM 序号把“第 N 个按钮”当成目标候选人按钮
- 每打一位候选人就默认切到聊天页确认，再回推荐页继续用旧索引
- 使用额外本机控制路径作为默认或兜底路径
