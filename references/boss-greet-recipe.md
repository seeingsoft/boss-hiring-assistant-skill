# BOSS 打招呼固定方案

## 定位

这个文档只处理“对已批准候选人发送首条站内招呼消息”。

它不是通用聊天方案，也不是自由生成方案。
目标是把“打招呼”收敛成最少动作、最少试错、最少风控暴露的固定路径。

## 前置条件

开始打招呼前，必须同时满足：

1. 候选人已经通过筛选，或已获用户批准进入沟通
2. 当前 Boss 登录态稳定
3. 当前任务正在使用 `web-access`
4. 当前线程可被正常打开
5. 当前话术已经准备好，不在页面里边想边写

## 固定主路径

1. 在当前页面定位目标候选人卡片
2. 用稳定标识绑定“候选人卡片”和“该卡片内的打招呼按钮”
3. 做第一次候选人身份确认
4. 点击该候选人卡片内已绑定的打招呼按钮
5. 在当前页面优先完成成功确认
6. 成功后记录发送结果和发送时间
7. 再进入下一位候选人

这是一条“一人一事务”路径。
不要先识别一批候选人，再回头按按钮序号批量点击。

## 稳定定位规则

推荐页打招呼时，默认禁止使用纯位置索引定位，例如：

- XPath 第 `N` 个按钮
- 第 `N` 张卡片里的按钮
- `button[4]`
- “当前列表第 2 个/第 4 个按钮”

必须优先使用：

1. 候选人唯一 ID
2. 卡片节点上的稳定属性，例如 `data-id`、`href`、业务主键
3. 候选人姓名 + 职位 + 当前卡片文本上下文的组合绑定

也就是说，先锁定“这是哪位候选人”，再锁定“这个按钮属于这位候选人”。
不要倒过来先点按钮，再猜它属于谁。

## 推荐页专用规则

如果当前动作发生在 `recommended_feed`：

1. 不要依赖卡片序号
2. 不要依赖按钮序号
3. 不要默认每打一人就 `navigate` 去聊天页做外部确认
4. 优先在当前推荐页完成成功确认

因为推荐页常带虚拟滚动和重渲染：

- 滚动后节点可能复用
- 返回页面后节点顺序可能变化
- 原来的 XPath 或按钮索引可能已不再对应原候选人

所以推荐页里最重要的不是“按钮还能不能点”，而是“这个按钮是不是仍然属于目标候选人”。

## 发送按钮的唯一点击路径

发送按钮默认只允许按下面顺序尝试：

1. `clickAt`
2. CDP `Input.dispatchMouseEvent`
3. 单次 DOM `element.click()` 兜底
4. 若按钮点击层失败，再补一次 Enter

任何一步成功后都要立即停止继续试其他发送变体。

## 成功确认

成功确认优先级固定为：

1. 当前推荐页卡片状态变化，例如：
   - 出现“已沟通 / 继续沟通 / 已打招呼”之类状态
   - 按钮文案发生变化
   - 卡片操作区状态变化
2. 当前页出现明确成功反馈，例如 toast 或状态提示
3. 只有当前页没有足够信号时，才允许做一次外部确认
4. 外部确认优先检查聊天列表或线程状态，不要默认反复跳页

不要只因为“点过发送按钮”就当作发送成功。

## 当前页成功信号的固定检查顺序

为了减少 `navigate` 和页面重渲染带来的索引失真，推荐页打招呼后，默认必须先在当前页检查下面这些信号：

1. 当前候选人卡片内按钮文案是否变化
   - 例如从“立即沟通 / 打招呼”变成“继续沟通 / 已沟通”之类状态
2. 当前候选人卡片操作区是否变化
   - 例如按钮消失、按钮禁用、按钮样式变化、操作区被替换
3. 当前候选人卡片主状态文案是否变化
   - 例如新增“已联系”“已沟通”“继续沟通”类标签
4. 页面是否出现短时成功提示
   - 例如 toast、轻提示、顶部提示条

只有当上面 4 类当前页信号都拿不到明确结论时，才允许离开推荐页做一次外部确认。

## 当前页成功信号的低 token 读取要求

读取当前页成功信号时，优先使用最小读取集合：

1. 目标候选人卡片文本
2. 目标候选人卡片内按钮文本
3. 目标候选人卡片操作区状态
4. 页面短时提示节点

不要为了确认一个候选人是否已打招呼，而重新读取整页所有卡片的完整结构。

优先输出结构化确认结果，例如：

- `button_text_changed=true`
- `card_status_changed=true`
- `toast_found=false`
- `greet_confirmed=true`

不要在这一层输出长篇解释。

## 外部确认的边界

如果必须离开推荐页做外部确认：

1. 只允许做一次
2. 确认完成后，如果还要继续处理下一位候选人，必须重新回到推荐页重新定位
3. 回到推荐页后，禁止复用上一轮的 XPath 索引、按钮序号、卡片序号

也就是说，一旦离开推荐页再回来，就视为列表可能已经重渲染，必须重新绑定目标候选人。

## 单轮发送上限

打招呼默认必须分轮执行。

单轮内必须遵守：

1. 小批量发送
2. 相邻发送之间加入随机短暂停顿
3. 每发送 2 到 3 人，做一次轻量状态校验
4. 单轮达到上限后立即暂停并汇报

不要因为同一套路径刚刚成功，就无限连续发送。

这里的“轻量状态校验”应优先检查：

- 当前页面是否仍在推荐页
- 当前列表是否仍正常可读
- 最近一位候选人的成功确认是否明确

不要把轻量校验做成一次完整的大诊断。

## 失败处理

一次候选人打招呼失败时，只允许：

1. 快速定位失败层级
2. 在同一层做一次有限补试
3. 仍失败则记录为 `greet_send_failed`
4. 进入可恢复暂停态或跳过该候选人，继续后续已批准候选人

不要因为一个候选人失败，就在页面上长时间研究前端事件系统。

如果失败层级属于“候选人绑定关系失效”，例如：

- 当前按钮已无法确认属于目标候选人
- 页面重渲染后卡片顺序变化
- 返回推荐页后原卡片索引失真

则不要继续点旧按钮。
必须先重新定位并重新绑定候选人，再决定是否补试。

## token 经济性要求

推荐页打招呼默认要优先节省 token：

1. 优先在单个 eval 中提取“候选人标识 + 卡片文本 + 按钮状态”
2. 优先在当前页完成成功确认，减少 `navigate`
3. 不要为每位候选人重复跑整页深度分析
4. 每位候选人只允许一次快速补试，不允许长链试错
5. 失败后输出结构化结果，不要输出长篇泛化原因分析

## 禁止项

以下行为不允许作为默认打招呼路径：

- 一边看页面一边临时生成长话术
- 起子代理研究 Vue 或前端事件
- 反复切换 DOM click / 坐标点击 / 截图方案
- 发送失败后长时间停在同一候选人页面试错
- 要求用户手动帮忙点击候选人或手动发送
- 用 XPath / DOM 序号把“第 N 个按钮”当成目标候选人按钮
- 每打一位候选人就默认切到聊天页确认，再回推荐页继续用旧索引
