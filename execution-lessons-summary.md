# 执行经验汇总

## 1. Boss 浏览器接管的首要原则

- 必须先让用户在真实 Chrome / Chromium 中登录 Boss 招聘者页面
- 必须先复用用户当前已登录的 Boss tab
- 不要通过 `web-access /new` 新开 Boss tab 再进入后台
- 不要默认把 Boss 浏览器控制下放给子代理
- `targetId` 变化时，先自动重新扫描 `/targets` 并重绑 Boss target，不要立即要求用户确认登录

## 2. 浏览器工具边界

- Boss 页面默认只允许使用 `web-access`
- `browser-use` 等会抢占浏览器控制的 skill，应先由龙虾代为禁用或移除
- `web-access` 的安装来源固定为：`https://github.com/eze-is/web-access`
- 默认禁止截图、OCR、图像理解读取 Boss 页面
- 只有 `web-access` 在同一路径失败并补试 2 次后，且用户明确同意，才允许截图作为最终故障定位手段

## 3. eval 的稳定写法

- 使用复杂 eval 前，必须先做最小探针
- 传参优先使用 `Content-Type: text/plain` + `--data-raw`
- 如果 eval 连续失败，先检查请求体编码、转义和内容类型，不要先怀疑页面结构

## 4. JD 读取

- 职位管理页实际内容在 iframe 中
- 读取在线 JD 时，优先访问：`https://www.zhipin.com/web/frame/job/list-new`
- 不要只访问 `/web/chat/job/list`

## 5. 在线简历读取

- Boss 聊天页是左右分栏 SPA
- 候选人简历常在 `/web/chat/index` 右侧面板中，不一定跳独立 URL
- 正确路径通常是：左侧点击候选人卡片，右侧读取简历面板 DOM

## 6. 筛选对象的业务边界

- 新一轮筛选默认优先处理未读 / 未处理候选人
- `inbound_chat` 以页面未读信号为主，任务记忆为辅
- `recommended_feed` 与 `search_results` 以任务记忆为主，页面信号为辅
- 不要把“当前可见候选人”直接等价成“本轮应处理候选人”
- 对推荐页，如果候选人卡片已经稳定显示，就直接读取当前 DOM，不要默认等待很久、先做截图快照分析，或误判为“页面未完全加载”

## 7. policy 缺失时的默认路径

- 缺少 `company-policy.yaml` / `approval-policy.yaml` 时，不要先问用户口述标准
- 正确顺序是：先读在线岗位和 JD，再生成默认 policy，然后只向用户确认差异项

## 8. Boss 发消息的稳定路径

- 打开线程、身份校验、输入、发送、成功确认是分层动作，不能混成一个模糊动作
- 发送前必须做三次候选人身份校验，避免把给 A 的消息发给 B
- 对“索要附件简历”的候选人，发送后必须继续跟进是否已发送附件，以及是否完成接收

## 9. Boss 发消息的风控经验

- 批量打招呼是 Boss 风控高敏感动作
- 默认不要把 DOM `btn.click()` 当成发送主路径
- 发送按钮优先使用 `web-access clickAt`
- 其次使用底层 CDP `Input.dispatchMouseEvent`
- DOM `element.click()` 只允许作为单次兜底
- 批量发送时必须小批量分轮，并在相邻发送间加入短暂停顿
- 一旦出现验证码、异常验证、登录态波动、页面跳转异常或失败率升高，立即停止整轮发送
- 推荐页批量打招呼时，真正的主问题往往不是“按钮点没点出去”，而是“点到的按钮是否仍属于目标候选人”
- 不要使用 XPath / DOM 序号去点第 N 个打招呼按钮
- 不要每打一人就跳聊天页确认再回推荐页复用旧索引
- 推荐页应优先在当前页完成成功确认，减少跳页和重渲染带来的定位失真
- 当前页成功确认应优先检查：按钮文案变化、卡片状态变化、操作区变化、toast 提示
- 如果推荐页卡片没有唯一 ID，先用 eval 按姓名和上下文找到卡片，再给卡片内按钮写临时 `data-*` 标记，最后用 CSS selector 调 `clickAt` 或 `click`
- 如果前几个推荐牛人打招呼成功、后续连续失败，并且用户手动点击也无反应，应优先判断为当日推荐牛人沟通权益耗尽，而不是 selector、Vue 事件或 click 方式失败
- 推荐页打招呼失败时，不要发散到前台真实鼠标点击；应优先检查权益/额度提示，疑似耗尽时进入 `paused_for_boss_contact_quota_exhausted`

## 10. 失败处理

- 强约束动作失败后，应进入可恢复暂停态
- 不要长时间递归遍历 DOM、坐标乱点、反复试多套前端方案
- 先走低 token 的分层诊断，再决定是否暂停或请求用户确认页面状态

## 11. 上下文膨胀风险

- 当龙虾长时间探索页面、输出大段泛化分析时，容易出现上下文膨胀
- 上下文接近或超过模型窗口时，更容易忘记 skill 中的硬规则，退回成通用浏览器助手模式
- 因此推荐页读取、首条打招呼、发送失败诊断都要使用短 recipe，而不是长篇推理
